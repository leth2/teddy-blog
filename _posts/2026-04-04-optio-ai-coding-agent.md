AI 코딩 에이전트가 실제로 쓸 만해지면서 새로운 문제가 생겼다. 에이전트는 코드를 쓸 수 있다. 그런데 그 코드를 어떻게 리뷰받고, 테스트하고, 배포까지 이어지게 할 것인가? 사람이 여전히 중간에 개입해야 하는가?

[Optio](https://github.com/jonwiggins/optio)는 이 질문에 대한 구체적인 대답이다. "태스크 제출부터 PR 머지까지" — 사람의 개입 없이, AI 에이전트가 전체 사이클을 돌리는 시스템이다. 단순한 코드 생성 도구가 아니라, **AI 에이전트를 위한 CI/CD 파이프라인**이다.

이 포스트에서는 Optio의 작동 방식을 단계별로 분해하고, 기존 CI/CD와 무엇이 근본적으로 다른지, 그리고 실제 도입을 고려한다면 무엇을 생각해야 하는지를 다룬다.

---

## Optio가 해결하는 문제

### 기존 "AI 코딩" 워크플로우의 한계

지금 대부분의 팀이 AI 코딩 에이전트를 쓰는 방식을 생각해보자:

1. 개발자가 Claude Code / Codex를 실행한다
2. 에이전트가 코드를 생성한다
3. 개발자가 결과를 검토하고 수정한다
4. 개발자가 PR을 올린다
5. 리뷰어가 리뷰한다
6. CI를 통과하면 머지한다

에이전트가 코드를 쓰는 건 자동화됐지만, 나머지는 여전히 사람이 한다. 에이전트는 "더 빠른 타이피스트" 수준에 머문다.

Optio의 질문은 이렇다: **왜 에이전트가 PR을 올리고, CI를 보고, 실패하면 스스로 고치고, 리뷰 코멘트를 반영하는 것까지 못 하는가?**

기술적으로는 가능하다. 그걸 시스템으로 만든 게 Optio다.

## Optio의 전체 플로우: 단계별 분석

### Step 1: 태스크 제출

Optio는 다양한 진입점을 지원한다:
- 웹 UI 직접 입력
- GitHub Issue
- Linear / Jira / Notion 연동

이 다양성이 중요하다. 팀의 기존 워크플로우를 바꾸지 않아도 된다. 개발자가 Jira에 이슈를 만들면, 그것이 Optio의 태스크가 된다. 백로그 관리 방식 그대로, 실행만 AI가 한다.

### Step 2: 격리된 실행 환경 프로비저닝

태스크가 들어오면 Optio는 **Kubernetes 파드를 해당 레포지토리 전용으로 생성**한다. 그리고 git worktree를 마운트한다.

이게 "pod-per-repo" 아키텍처다. 이 부분이 Optio의 핵심 기술 결정 중 하나다. 뒤에서 자세히 다룬다.

### Step 3: AI 에이전트 실행

격리된 환경에서 AI 에이전트가 코드를 작성한다. 지원하는 에이전트:
- **Claude Code** (Anthropic)
- **Codex** (OpenAI)
- **GitHub Copilot**

에이전트는 레포지토리의 맥락을 갖고, 태스크 설명을 인풋으로 받아, 코드를 생성하고 커밋한다.

### Step 4: PR 오픈 및 CI 모니터링

에이전트가 코드 작성을 완료하면 Optio가 자동으로 PR을 연다. 그리고 **30초마다 CI 상태를 폴링**한다.

여기서 중요한 건 Optio가 단순히 기다리지 않는다는 점이다. CI 결과에 따라 다음 액션이 결정된다.

### Step 5: 피드백 루프 — 이게 핵심이다

Optio의 진짜 차별점은 여기서 나온다.

#### 시나리오 A: CI 실패

```
CI 실패 감지
→ 실패 컨텍스트 수집 (에러 로그, 실패한 테스트, 스택트레이스)
→ 에이전트 재개 (with 실패 컨텍스트)
→ 에이전트가 실패 원인 분석 및 수정
→ 새 커밋 푸시
→ CI 다시 실행
→ 반복 (통과할 때까지)
```

사람이 했다면: CI 실패 알림 받기 → 로그 확인 → 원인 파악 → 수정 → 재푸시. 최소 30분, 맥락 전환 비용 포함하면 더 길다.

Optio는 이걸 자동으로, 잠 안 자고, 반복한다.

#### 시나리오 B: 리뷰어 코멘트

```
리뷰어가 PR에 변경 요청 코멘트 작성
→ Optio가 코멘트 감지
→ 코멘트 내용을 에이전트에게 전달
→ 에이전트가 코멘트 분석 및 코드 수정
→ 새 커밋 푸시
→ 리뷰어에게 수정 완료 알림
```

이건 단순한 자동화가 아니다. AI 에이전트가 **코드 리뷰 피드백을 이해하고 반영**한다는 의미다. 물론 복잡한 아키텍처 결정이나 주관적 의견은 여전히 사람이 판단해야 하지만, 명확한 기술적 피드백은 에이전트가 처리할 수 있다.

#### 시나리오 C: 모든 것 통과

```
CI 통과 + 리뷰어 승인
→ Squash merge 실행
→ 이슈 자동 Close
→ 파드 종료 및 리소스 정리
```

태스크 시작부터 이슈 닫힘까지 완전 자동화.

### Step 6: 코드 리뷰 에이전트 (별도 서브태스크)

흥미로운 부분이 있다. Optio는 **코드 리뷰 에이전트를 별도 서브태스크로 실행**한다. 코드를 쓰는 에이전트와, 그 코드를 리뷰하는 에이전트가 분리되어 있다.

왜 이렇게 설계했을까?

단일 에이전트가 자기 코드를 리뷰하면 자기 편향이 생긴다. "내가 쓴 게 맞겠지"라는 편향. 분리된 에이전트는 이 편향 없이 코드를 검토한다. 인간 팀에서 "작성자가 자기 코드를 리뷰하지 않는다"는 원칙과 같다.

## 기존 CI/CD와 근본적으로 무엇이 다른가

### 전통적 CI/CD 패러다임

전통적인 CI/CD는 **코드는 인간이 쓴다**는 가정 위에 세워졌다. 파이프라인은 인간이 쓴 코드를 받아서 테스트하고 배포한다. 파이프라인이 실패하면 인간에게 알린다. 인간이 고친다.

```
인간 → [코드 작성] → CI/CD → [테스트/배포] → 실패 시 → 인간에게 알림 → 인간이 고침
```

이 루프에서 AI는 기껏해야 "더 빠른 코드 작성 도구"다.

### Optio의 패러다임

Optio는 이 루프에서 **인간을 선택적 참여자**로 만든다.

```
인간 → [태스크 정의] → Optio → [에이전트 실행 + CI 루프] → 실패 시 → 에이전트가 고침 → 통과 시 → 인간에게 승인 요청
```

인간은 처음(무엇을 할 것인가)과 끝(이게 맞는가)에만 등장한다. 중간의 실행-테스트-수정 루프는 자동화된다.

이게 근본적인 패러다임 전환이다. CI/CD는 더 이상 "자동화된 테스트"가 아니라 **"AI 에이전트의 실행 환경"**이 된다.

### 전통적 파이프라인과 비교

| 항목 | 전통적 CI/CD | Optio |
|------|------------|-------|
| 코드 작성 주체 | 인간 | AI 에이전트 |
| CI 실패 처리 | 인간에게 알림 | 에이전트가 자동 수정 |
| 리뷰 코멘트 처리 | 인간이 반영 | 에이전트가 반영 |
| 실행 환경 | 공유 또는 고정 | 태스크별 격리 파드 |
| 인간 개입 시점 | 전 과정 | 태스크 정의 + 최종 승인 |

## Pod-per-Repo 아키텍처: 의미와 트레이드오프

### 왜 파드를 레포지토리별로 격리하는가

Optio는 각 태스크마다 독립적인 Kubernetes 파드를 생성한다. 이 결정에는 중요한 이유가 있다.

**1. 보안 격리**

AI 에이전트는 코드를 실행한다. 악의적이거나 버그 있는 코드가 다른 레포지토리나 시스템에 영향을 주면 안 된다. 파드 격리는 "폭발 반경(blast radius)"을 태스크 하나로 제한한다.

**2. 환경 재현성**

레포지토리마다 다른 언어 버전, 의존성, 환경 변수가 있다. 격리된 파드는 각 레포지토리의 정확한 환경을 재현한다. "내 로컬에서는 됐는데" 문제가 없다.

**3. 병렬 처리**

여러 태스크가 동시에 진행될 수 있다. 파드가 격리되어 있으므로 서로 영향을 주지 않는다. 5개 이슈를 동시에 처리해도 각 에이전트는 독립적으로 작동한다.

**4. 상태 관리**

git worktree를 파드에 마운트하면 파드 생명주기 = 태스크 생명주기가 된다. 태스크가 완료되면 파드를 종료하고 리소스를 정리한다. 남은 찌꺼기가 없다.

### 트레이드오프

물론 이 아키텍처에는 비용이 따른다.

**프로비저닝 지연**

파드를 생성하고 환경을 세팅하는 데 시간이 걸린다. 작은 태스크라도 수십 초의 준비 시간이 필요하다. 기존 환경을 재사용하는 방식보다 느리다.

**Kubernetes 의존성**

Kubernetes 클러스터가 없으면 Optio를 운영할 수 없다. 소규모 팀이나 클라우드 인프라가 없는 팀에게는 진입 장벽이다.

**비용**

파드가 생성될 때마다 컴퓨팅 리소스가 사용된다. 태스크가 많아질수록 클러스터 비용이 올라간다. AI 에이전트 API 비용 + Kubernetes 비용 + 개발자 시간 절약 비용을 비교해야 한다.

**복잡성**

태스크 실패 시 파드 로그를 봐야 한다. 파드가 이미 종료됐으면 로그를 보존하는 메커니즘이 필요하다. 인프라 운영 복잡도가 올라간다.

## Optio의 아키텍처: 기술 스택 분석

### 컴포넌트 구성

```
┌─────────────────────────────────────────┐
│  Web UI (Next.js)                        │
│  - 태스크 제출, 상태 조회, 비용 대시보드  │
└──────────────────┬──────────────────────┘
                   │ HTTP
┌──────────────────▼──────────────────────┐
│  API Server (Fastify)                    │
│  - 태스크 관리, 인증, 이벤트 처리         │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  Worker Queue (BullMQ + Redis)           │
│  - 태스크 큐잉, 재시도, 우선순위         │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  Kubernetes                              │
│  - Pod-per-repo 프로비저닝               │
│  - 에이전트 실행                          │
└─────────────────────────────────────────┘
          │ 상태 저장
┌─────────▼───────────────────────────────┐
│  Postgres                                │
│  - 태스크 히스토리, 비용 데이터           │
└─────────────────────────────────────────┘
```

각 컴포넌트의 선택이 흥미롭다.

**Fastify** — Node.js 기반이지만 Express보다 훨씬 빠른 웹 프레임워크. AI 에이전트 이벤트가 빈번하게 들어오는 환경에서 성능이 중요하다.

**BullMQ** — Redis 기반 작업 큐. 태스크를 큐에 넣고 워커가 처리한다. 태스크 실패 시 재시도, 우선순위 조절, 지연 실행 등이 모두 지원된다. CI 피드백 루프처럼 반복적인 실행에 적합하다.

**Postgres** — 태스크 히스토리와 비용 데이터를 영구 저장. Redis는 휘발성이라 영속성이 필요한 데이터는 Postgres에.

### 실시간 대시보드와 비용 분석

Optio는 실시간 대시보드를 제공한다. 주목할 점은 **비용 분석(cost analytics)**이 포함되어 있다는 것이다.

AI 에이전트를 사용하면 API 호출 비용이 발생한다. 태스크마다 얼마나 썼는지, 어떤 타입의 태스크가 더 비싼지, 月 비용 추세는 어떤지 — 이게 보이지 않으면 AI 도입의 ROI를 계산할 수 없다.

비용 투명성을 처음부터 설계에 포함한 것은 성숙한 접근이다.

## 도입 시 고려사항

### 보안

AI 에이전트가 코드 레포지토리에 직접 접근하고 커밋한다. 이것은 중요한 권한이다.

**체크리스트:**
- 에이전트가 접근하는 레포지토리 범위를 최소화했는가?
- 파드의 네트워크 정책을 설정했는가? (외부 인터넷 접근 제한)
- 에이전트 API 키가 파드 안에서만 유효한가?
- 악의적인 태스크 제출을 막는 인증이 있는가?
- 에이전트가 생성한 코드에 비밀 정보(API 키, 비밀번호)가 포함될 가능성을 차단했는가?
- PR 강제 머지를 막는 Branch Protection Rule이 있는가?

특히 마지막 항목이 중요하다. 에이전트가 완전 자동으로 머지하는 시나리오라면, 브랜치 보호 설정이 유일한 방어선이 된다.

### 비용

Optio의 비용 구조는 세 계층이다:

1. **AI 에이전트 API 비용** — 태스크당 몇 센트 ~ 몇 달러. 복잡한 태스크, 긴 컨텍스트일수록 비쌈.
2. **Kubernetes 클러스터 비용** — 파드 프로비저닝 및 실행 비용. 클라우드 프로바이더마다 다름.
3. **기회 비용** — 개발자 시간이 절약되는 것 vs. 인프라 운영 오버헤드.

소규모 팀(5명 이하)에서는 Kubernetes 운영 복잡도 vs. 절약 시간의 비율이 안 맞을 수 있다. 중규모 이상(10명+), 특히 반복적인 태스크가 많은 팀에서 ROI가 나온다.

**비용 최적화 팁:**
- 단순하고 반복적인 태스크를 먼저 Optio로 전환한다 (리팩토링, 테스트 추가, 문서화)
- 태스크당 에이전트 실행 시간 제한을 설정한다
- 비용 대시보드를 주기적으로 리뷰해 비용이 높은 태스크 타입을 파악한다

### 신뢰도

"AI가 쓴 코드가 프로덕션에 들어가도 되는가?" 이 질문이 핵심이다.

Optio는 이 문제를 몇 가지 방식으로 다룬다:

1. **CI 게이트** — 테스트를 통과하지 않으면 머지되지 않는다. 테스트 커버리지가 충분하다면 실질적인 안전망이 된다.
2. **코드 리뷰 에이전트** — 별도 에이전트가 리뷰하지만, 결국 AI가 AI를 검토하는 것이다. 한계가 있다.
3. **인간 승인** — 최종 머지 전 인간 승인을 필수로 설정할 수 있다.

현실적인 권장사항: 처음에는 **인간 승인 필수 모드**로 시작하라. 에이전트의 판단을 신뢰하게 되는 데이터를 쌓고, 그 이후에 자동 머지 범위를 점진적으로 넓혀가라.

리스크 허용 범위에 따라:
- **저위험**: 문서 업데이트, 린트 수정, 테스트 추가 → 자동 머지 가능
- **중위험**: 기능 구현, 리팩토링 → 인간 리뷰 후 머지
- **고위험**: 보안 관련, 데이터 마이그레이션, 외부 API 변경 → 절대 자동 머지 금지

## 기존 CI/CD와의 비교: 사고 실험

당신의 팀에 이런 태스크가 있다고 하자: "User API의 모든 엔드포인트에 rate limiting 추가"

**기존 방식:**
1. 개발자가 태스크 픽업 (1일 대기)
2. 기존 코드 파악 (1~2시간)
3. 구현 (2~3시간)
4. 테스트 작성 (1~2시간)
5. PR 생성 (30분)
6. 리뷰 대기 (0.5~2일)
7. 리뷰 반영 (1시간)
8. 머지 (30분)

총: 2~4일

**Optio 방식:**
1. 이슈 생성 (5분)
2. Optio가 파드 생성 + 에이전트 실행 (30분~2시간)
3. PR 오픈, CI 통과
4. 인간 리뷰 (30분~1시간)
5. 승인 후 자동 머지 (즉시)

총: 1~4시간

물론 에이전트가 완벽하지 않다. CI를 여러 번 실패하고 수정할 수도 있다. 하지만 그 시간에 개발자는 다른 일을 하고 있다.

## 현 시점에서의 Optio

Optio는 아직 초기 오픈소스 프로젝트다. 프로덕션 환경에서 완전 검증된 솔루션이라기보다는 "이 방향이 맞다"는 것을 증명하는 프로토타입에 가깝다.

그러나 이 방향 자체는 피할 수 없는 흐름이다. GitHub Copilot Workspace, Devin, SWE-agent — 모두 같은 방향을 가리키고 있다. 에이전트가 태스크를 받아서 PR까지 완성하는 흐름.

Optio의 가치는 이 흐름을 **self-hosted, 오픈소스, 커스터마이징 가능한 형태**로 제공한다는 것이다. 클라우드 벤더 종속 없이 같은 패러다임을 가져갈 수 있다.

## 핵심 요약

**Optio란:** 태스크 정의부터 PR 머지까지를 AI 에이전트가 자율적으로 처리하는 워크플로우 오케스트레이터

**핵심 혁신:**
- **피드백 루프**: CI 실패 및 리뷰 코멘트를 에이전트가 자동으로 처리
- **pod-per-repo**: 격리된 실행 환경으로 보안과 재현성 확보
- **비용 투명성**: AI 사용 비용을 태스크 단위로 추적

**기존 CI/CD와의 차이:**
- CI/CD: 인간이 쓴 코드를 자동으로 검증
- Optio: AI가 쓴 코드를 AI가 검증하고 수정, 인간은 승인만

**도입 권장 순서:**
1. 저위험 반복 태스크부터 시작
2. 인간 승인 필수로 운영하며 신뢰 데이터 축적
3. 비용/시간 절약 ROI 검증 후 확대

## 다음에 생각해볼 것들

1. **우리 팀의 반복 태스크 중 Optio에 적합한 것은 무엇인가?** 테스트 추가, 문서화, 린트 수정, 소규모 리팩토링?

2. **"에이전트가 쓴 코드"의 신뢰 기준을 어떻게 정할 것인가?** 테스트 커버리지? 코드 복잡도 한계치? 특정 파일/디렉토리 제외?

3. **pod-per-repo 대신 더 가벼운 격리 방법은 없는가?** Docker Compose? Firecracker microVM? 팀 규모에 따른 적정 수준은?

4. **코드 리뷰 에이전트의 품질을 어떻게 검증할 것인가?** 에이전트가 놓치는 패턴을 사람이 캐치하는 메커니즘이 필요하다.

5. **Optio와 SDD/spec-validator를 연결하면 어떻게 될까?** 스펙이 있으면 에이전트는 그것을 구현 기준으로 삼고, spec-validator는 구현이 스펙을 충족하는지 검증하는 CI 게이트가 된다. 태스크 → 스펙 → 구현 → 검증의 완전 자동화 루프.
