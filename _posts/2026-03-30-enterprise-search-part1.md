> **이 글은 예비 연구다.** n=10 커밋, Zulip 단일 코드베이스.  
> "이것이 결론"이 아니라 "이런 패턴을 발견했다"는 기록이다.

---

## 왜 이 연구를 시작했나

AI가 작은 코드베이스에서 잘 동작하는 건 이미 증명됐다. Slop Score 낮은 스펙 + 에이전트 = 테스트 통과율 100%. 이건 우리가 직접 측정했다.

그런데 엔터프라이즈 레거시 코드베이스는 다르다.

```
파일 수:    수천~수만 개
문서화:     없거나 구식
AI 컨텍스트 윈도우: 코드 전체를 담을 수 없음
```

이 환경에서 AI가 "이 기능이 어떻게 동작하나?"나 "어떤 파일을 수정해야 하나?"를 답하려면 **검색 도구**가 필요하다. 그런데 어떤 도구가 얼마나 도움이 되는가? 정량적으로 측정한 연구가 드물다.

그래서 직접 실험했다.

---

## 실험 설계

**코드베이스:** Zulip (팀 채팅 오픈소스, Python 1987개 + TypeScript 459개, 202MB)

Zulip을 선택한 이유: 실제 엔터프라이즈가 사용하는 규모, Python + TypeScript 혼합, 문서화 불완전한 내부 로직 존재.

**실험한 검색 방식:**

| 이름 | 방식 |
|------|------|
| A1 | 파일 목록만 AI에게 전달 |
| A2 | ripgrep (텍스트 검색) |
| A3v2 | ripgrep + AST 심볼 + docstring |
| A5 | AI가 쿼리에서 키워드를 동적 생성 → ripgrep + AST |
| C | networkx 그래프DB (호출 관계, 연결 수 기반) |
| C-det | 그래프 + AI 선택 제거 (완전 결정론적) |
| B (RAG) | Chroma 벡터DB + sentence-transformers |
| Dv2 | 그래프(구조 파악) + A5(내용 읽기) 조합 |

**측정 지표:**
- 파일 예측 태스크: Precision / Recall / F1 (실제 커밋 변경 파일과 비교)
- 탐색 태스크: 답변 품질 (주관 평가 + 발견한 코드 구체성)
- 재현성: 동일 쿼리 5회 반복 Jaccard 유사도

## 결과 1: 파일 예측 정량 평가

10개 Zulip 커밋의 메시지를 쿼리로 사용해서 "어떤 파일이 수정됐는가?"를 예측했다.

**정답 데이터셋 (실제 커밋):**

| # | 커밋 메시지 요약 | 정답 파일 수 |
|---|----------------|------------|
| 1 | reminder: topic_link 생성 수정 | 3개 |
| 2 | populate_db: -o 플래그 버그 수정 | 1개 |
| 3 | event_queue: 오래된 큐 offline 처리 | 1개 |
| 4 | event_queue: 변수명 변경 리팩터링 | 1개 |
| 5 | mattermost: 다중 팀 변환 지원 | 2개 |
| 6 | remote-billing: 에러 메시지 수정 | 1개 |
| 7 | scim: 커스텀 프로필 필드 동기화 | 3개 |
| 8 | rate_limiter: Tornado 인메모리 제거 | 2개 |
| 9 | rate_limiter: Redis GCRA 업데이트 | 1개 |
| 10 | api-docs: 이메일 도메인 엔드포인트 문서화 | 3개 |

**전체 방식 비교 (F1 기준):**

| 방식 | Precision | Recall | F1 | 비고 |
|------|-----------|--------|----|------|
| **v3 (ripgrep + 테스트필터)** | **0.382** | **0.617** | **0.430** | 최고 |
| v4 (+ 전체 도메인맵) | - | - | 0.355 | v3보다 낮아짐 |
| v5 (+ 선택적 도메인맵) | - | - | 0.349 | v3보다 낮아짐 |
| C-det (그래프 결정론) | 0.233 | 0.500 | 0.292 | |
| 앙상블 (ripgrep + 그래프) | 0.300 | 0.233 | 0.250 | |
| AI 단독 (Zulip 힌트) | - | - | 0.211 | 도구 없음 |
| AI 단독 (힌트 없음) | - | - | 0.204 | 베이스라인 |
| co-change 히스토리 | - | - | 0.164 | |
| **RAG (Chroma 벡터DB)** | 0.093 | 0.217 | **0.120** | 가장 낮음 |
| 기본 ripgrep (테스트 포함) | - | - | 0.057 | 테스트 파일 오염 |

**v3 커밋별 상세 (정답과 예측 비교):**

| 커밋 | F1 | 비고 |
|------|-----|------|
| populate_db | **1.0** | 파일명이 커밋 메시지에 직접 포함 |
| mattermost | **1.0** | 두 파일 모두 정확 예측 |
| event_queue × 2 | 0.5 | 정답 포함하나 불필요 파일도 포함 |
| scim | 0.4 | 부분 성공 |
| rate_limiter × 2 | 0.29~0.33 | recall 높으나 precision 낮음 |
| reminder | 0.29 | topic_link_util 맞췄으나 나머지 실패 |
| remote-billing | **0** | 정보 한계 (billing → push_notifications 매핑 불가) |
| api-docs | **0** | 정보 한계 (api-docs → openapi/ 약어 매핑 불가) |

**첫 번째 발견: 도구의 기여는 실제다.**

AI 단독(F1=0.211) vs 도구 포함 v3(F1=0.430). 두 배 차이.

이건 통제 실험으로 검증했다. Zulip은 공개 프로젝트라 Claude가 학습 데이터로 알고 있을 가능성이 있다. 그래서 "Zulip 힌트 없이" 테스트했을 때도 F1=0.204로 거의 같았다. AI 내부 지식 단독으로는 F1=0.2 수준이고, ripgrep 도구를 더하면 0.43이 된다.

ripgrep이 하는 역할: AI의 추측을 실제 존재하는 파일로 끌어당기는 것. AI가 `zerver/lib/reminder.py`라고 만들어낸 경로를 실제 파일인 `zerver/lib/reminders.py`로 고정시키는 효과다.

**두 번째 발견: 더 많은 컨텍스트가 항상 좋은 건 아니다.**

직관적으로는 도메인 지식(파일별 docstring, 디렉토리 구조)을 AI에게 더 많이 주면 좋아질 것 같다. 실험 결과는 반대였다.

```
v3 (기본 ripgrep):          F1=0.430
v4 (+ 전체 도메인맵):       F1=0.355  ← 낮아짐
v5 (+ 선택적 도메인맵):     F1=0.349  ← 낮아짐
```

왜? Claude는 Zulip(공개 프로젝트)에 대한 내부 지식을 이미 갖고 있다. 우리가 제공한 docstring이 그 내부 지식과 충돌하거나 교란시켰다.

"이미 아는 것에 설명을 더 붙이면 오히려 혼란스러워진다."

**세 번째 발견: 그래프는 파일 예측에 약하다.**

C-det(그래프 결정론)의 F1=0.292. v3(0.430)보다 낮다.

이유: 연결 수(degree)가 높은 파일 = 코드베이스에서 중요한 파일. 하지만 **"중요한 파일 ≠ 수정할 파일"**.

예) `rate_limiter: Remove Tornado in-memory backend` 커밋의 정답은 `rate_limiter.py`. 그런데 그래프에서 "Tornado"와 연결된 고연결 파일들은 `tornado/event_queue.py`, `tornado/views.py`였다.

## 결과 2: 탐색 태스크 주관 평가

> 탐색 태스크의 정량화 방법은 아직 설계 중이다. 이번에는 "동일 쿼리에서 각 방식이 발견한 코드의 구체성"으로 평가했다.

**탐색 쿼리:** "메시지 전송 시 알림 처리 경로는 어떻게 되는가?"  
**기능추가 쿼리:** "새로운 알림 채널 SMS를 추가하려면 어디에 코드를 넣어야 하나?"  
**디버깅 쿼리:** "푸시 알림이 특정 유저에게만 안 가는 버그, 어디를 봐야 하나?"

**쿼리 유형별 성능 요약:**

| 방식 | 탐색 | 기능추가 | 디버깅 |
|------|------|---------|--------|
| A1 (파일목록만) | 추론만, 내용 없음 | ❌ | ❌ |
| A3v2 (고정 키워드) | ✅ 구현 세부 추적 | ❌ 엉뚱한 파일 | ❌ 엉뚱한 파일 |
| A5 (동적 키워드) | ✅ | ✅ notification_data 위치 정확 | ✅ 체크리스트 제시 |
| C (그래프) | ✅ 아키텍처 구조 | ❌ 일반론 | ❌ 일반론 |
| **Dv2 (그래프+A5)** | **✅✅ 가장 깊음** | ✅ | ✅ |

**Dv2 (그래프 + 적응형 ripgrep)가 가장 깊은 인사이트를 냈다.**

그래프가 `queue.py`를 핵심 파일로 식별했고, 거기서 다른 방식이 놓친 내용을 찾았다:

```python
# zerver/lib/queue.py::queue_event_on_commit
def queue_event_on_commit(queue_name: str, event: dict) -> None:
    transaction.on_commit(
        lambda: queue_json_publish_rollback_unsafe(queue_name, event)
    )
```

"왜 알림을 워커 큐로 처리하는가?"의 답이 여기 있었다 — DB 트랜잭션 커밋 후 발행. 사용자 요청을 즉시 반환하면서 알림은 백그라운드 처리.

A3v2(ripgrep + AST)는 구현 세부사항(FCM/APNs 전송 로직)을 더 잘 추적했다. 그래프와 ripgrep이 서로 다른 강점을 가진다.

**쿼리 유형별 성능 차이:**

"알림 처리 경로"(탐색) 쿼리에서는 모든 방식이 잘 작동했다. 하지만 "SMS 알림 추가하려면 어디에 코드를 넣어야 하나?"(기능추가) 쿼리에서 A3v2가 실패하고 A5(동적 키워드)가 성공했다.

이유: A3v2의 검색 키워드가 하드코딩(`push_notification`, `missedmessage` 등)되어 있어서 쿼리 유형이 달라지면 엉뚱한 파일을 찾는다. A5는 AI가 쿼리에서 키워드를 동적으로 생성해서 적응한다.

## 결과 2-1: RAG는 왜 가장 낮은가

RAG(Chroma 벡터DB)의 F1=0.120, P=0.093, R=0.217. 8가지 방식 중 최하위.

기대를 깬 결과다. "의미론적 검색이니까 더 잘 할 것"이라고 예상했는데 반대였다.

**이유:** 벡터 유사도 = "의미적으로 유사한 코드"를 찾는다. 하지만 "어떤 파일이 수정됐는가?"는 "의미 유사"가 아니라 "실제 구현 위치"를 물어보는 것이다.

예시:
```
커밋: "event_queue: Mark long-lived queues offline and send notifications"
RAG가 찾은 것: worker/missedmessage_mobile_notifications.py (의미 유사)
실제 정답:     tornado/event_queue.py (구현 위치)
```

알림 관련 코드를 다 찾았는데, 이 커밋에서 실제로 수정된 파일은 event_queue였다. "주제 유사성"과 "수정 대상"이 다른 것이다.

**RAG가 실제로 강한 것:** 탐색 태스크에서 Python과 TypeScript를 함께 검색하는 크로스 언어 검색. 이번 실험의 파일 예측 태스크는 RAG의 강점을 측정하지 못했다.

## 결과 3: 재현성

동일 쿼리를 5회 반복했을 때 결과가 얼마나 일관되는가.

| 방식 | Jaccard 유사도 | 해석 |
|------|--------------|------|
| C-det (그래프 결정론) | **1.000** | 완전 동일 |
| A5 (적응형) | 0.600 | 주의: 빈 결과끼리 유사도=1.0 착시 |
| A3v2 | 0.300 | 실질적으로 가장 안정 |
| C (그래프 + AI 선택) | 0.218 | 가장 불안정 |

그래프 자체는 결정론적이지만, AI가 선택 단계에서 Run마다 다른 파일을 고른다. AI 선택을 제거하면(C-det) 재현성이 1.0이 된다.

하지만 앞서 봤듯이 C-det의 파일 예측 F1은 낮다. **재현성과 정확도는 트레이드오프다.**

## 예측 불가능한 케이스의 실체

두 커밋은 모든 방식에서 F1=0이었다:

**케이스 1:** `remote-billing: Update error for user without has_billing_access`
- 정답: `zerver/views/push_notifications.py`
- 왜 실패: "billing" 관련 수정이 "push_notifications" 파일에 있다. Zulip의 remote server billing 로직이 push notification view에 구현되어 있다는 것을 코드베이스 외부 지식 없이는 알 수 없다.

**케이스 2:** `api-docs: Document endpoints for allowed email address domains`
- 정답: `zerver/openapi/` 디렉토리 3개 파일
- 왜 실패: "api-docs"가 `openapi/` 디렉토리를 의미한다는 매핑이 필요하다.

이 케이스들은 도구의 한계가 아니라 **정보의 한계**다. 코드베이스 전체 아키텍처를 이해한 사람만 예측할 수 있다.

현재 방식의 F1 상한은 0.430 정도다. 나머지 0.570을 올리려면 코드베이스 도메인 지식의 구조적 표현이 필요하다. 이건 다음 연구 주제다.

## 태스크별 최적 조합

| 태스크 | 최적 방식 | 이유 |
|--------|---------|------|
| "어떤 파일을 수정해야 하나?" | v3 (ripgrep + 테스트 필터) | 정량 F1 최고 |
| "이 코드가 어떻게 동작하나?" | Dv2 (그래프 + 적응형 ripgrep) | 구조 + 내용 조합 |
| "이 버그 어디서 고치나?" | A5 (동적 키워드 ripgrep) | 쿼리 적응성 |
| "빠른 구조 파악" | C-det (결정론적 그래프) | 재현성 1.0, 빠름 |
| "Python + TypeScript 함께" | RAG (Chroma) | 언어 경계 넘기 |

하나의 방식이 모든 태스크에서 최고가 아니다. 태스크에 따라 선택해야 한다.

## 마무리

"엔터프라이즈 코드베이스에서는 RAG나 그래프DB가 필요하다"는 직관이 있다.

이번 실험이 말하는 것:

> **그래프는 구조 탐색에 강하고, ripgrep은 수정 위치 찾기에 강하다. RAG는 언어 경계를 넘을 때 유일한 선택이다. 하지만 더 많은 컨텍스트가 항상 좋은 건 아니다.**

그리고 가장 단순한 접근(v3: ripgrep + 테스트 필터)이 정량적으로 가장 좋았다.

이건 엔터프라이즈 AI 엔지니어링의 시작이다. 다음 편에서 다른 코드베이스로 이 패턴이 재현되는지 확인한다.
