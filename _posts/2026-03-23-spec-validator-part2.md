## 1편에서 멈춘 곳

1편에서 Slop Score 알고리즘을 설계하고 Phase 1 규칙 기반 탐지기를 만들었다. 그런데 실제 스펙 파일에 적용하자 바로 문제가 드러났다.

`memo-app/tasks.md → S3: 23/100 → WARN`

태스크 문서에 Pre/Postcondition이 없어서 낮게 나온 건데, 태스크 문서가 Pre/Post를 쓸 이유가 없다. **S3가 모든 문서에 공통으로 적용되는 게 잘못이었다.**

그리고 더 근본적인 문제: 규칙으로 잡을 수 없는 것들이 있었다.

> "왜 Express인가?" "왜 SQLite인가?"

키워드로는 탐지 불가능하다.

## Phase 1.5: 문서 유형별 프로파일 분리

false positive의 원인을 추적하니 S3가 핵심이었다. design.md에서는 유효하지만 requirements/tasks에서는 구조적으로 낮게 나올 수밖에 없었다.

Signal을 문서 유형에 맞게 재구조화했다:

```
공통:           S1(결정밀도), S4(코드위장)
requirements:  S_AC (수락 기준 완결성)
design:        S3(계약 완결성) + S4(강화)
tasks:         S_DONE (완료 기준 + 의존성)
```

결과: tasks.md WARN → OK. S3 false positive 제거.

그리고 Agent Kill 게이트를 `sdd:spec-impl` 앞에 삽입했다. SLOP(Score ≥ 70)이면 에이전트 실행을 차단한다.

Auto-Fix도 설계했다. AI가 먼저 고치는 게 아니라 인간이 결정권을 쥔 채 제안을 받는 방식으로:

```
❌ AGENT KILL: Slop Score 74

이렇게 고치면 통과할 것 같습니다:
  REQ-003에 추가 제안:
    AC: "이메일 형식 검증 실패 시 400 반환"

수락하시겠습니까? [Y/n/편집]
```

## Phase 1.6: 구조화 EARS — Rolls-Royce에서 온 개념

테스트 중 흥미로운 발견이 있었다.

```
# 문장형 EARS → S_AC: 100
- AC-001: 유효한 요청 → 201 Created

# 구조화 EARS → S_AC: 100
1. WHEN an authenticated user submits a valid create request
   THEN the TODO API SHALL return 201 Created
```

점수는 같다. 그런데 **구조화 EARS만 기계가 파싱할 수 있다.**

`WHEN[조건] → THEN[시스템] SHALL[결과]` 패턴은 테스트 코드로 자동 변환 가능하다. 나중에 "Spec = TC" 원칙이 실제로 구현될 수 있는 형태다.

이건 2009년 Alistair Mavin이 Rolls-Royce에서 만든 EARS(Easy Approach to Requirements Syntax)의 원형이다. 항공우주/방산에서 요구사항 언어를 표준화한 이유와 AI 에이전트 스펙에 구조화 EARS가 필요한 이유가 같다: **모호한 언어는 검증이 불가능하다.**

5가지 템플릿:

| 유형 | 패턴 | 용도 |
|------|------|------|
| Ubiquitous | `The [시스템] SHALL [응답]` | 기본 요구사항 |
| Event-driven | `WHEN [트리거] SHALL [응답]` | 이벤트 발생 시 |
| Unwanted | `IF [에러] THEN SHALL [처리]` | 예외/에러 |
| State-driven | `WHILE [상태] SHALL [응답]` | 상태 유지 중 |
| Optional | `WHERE [포함 시] SHALL [응답]` | 선택적 기능 |

S_AC에 구조화 EARS 패턴 탐지를 추가했다. 멀티라인 `WHEN...THEN...SHALL` 쌍이 있을수록 점수가 올라간다.

## Phase 2.0: LLM Judge

규칙으로 못 잡는 신호들을 위해 cc-proxy(Claude Code 토큰 프록시 서버)를 통한 LLM Judge를 도입했다.

첫 번째 실제 결과:

```
design.md → S_WHY: 65/100 ⚠️

LLM Judge 판단:
  - Express vs Fastify/Koa 비교 없음
  - better-sqlite3 선택 이유 없음
  - 3계층 아키텍처 트레이드오프 없음
```

규칙 기반으로는 절대 탐지 불가능한 피드백이다. S1/S3/S4 점수가 모두 좋아도 "왜"가 없으면 슬롭이라는 판단을 LLM Judge가 해냈다.

단, LLM Judge는 환경에 따라 점수가 달라지는 재현성 문제가 있었다. 해결책: `--with-llm` 플래그로 분리. 기본 실행은 규칙 기반만 사용해 재현 가능한 점수를 보장한다.

## 실증 실험: 5개 데이터 포인트

todo-api를 5가지 품질의 스펙으로 구현해서 테스트 통과율을 측정했다.

**스펙 설계 방식:** Slop Score를 역산해서 각 목표 점수에 맞는 스펙을 만들었다. 좋은 스펙에서 요소를 제거하거나 나쁜 표현을 추가해 점수를 조정했다.

**결과:**

| 스펙 | Slop Score | 테스트 통과율 | 구현 갭 수 |
|------|-----------|------------|----------|
| 좋은 스펙 | 0 | 100% (25/25) | 7건 |
| 거의 좋은 스펙 | 19 | 96% (24/25) | 9건 |
| 중간 스펙 | 51 | 92% (23/25) | 9건 |
| 거의 나쁜 스펙 | 79 | 72% (18/25) | 20건 |
| 나쁜 스펙 | 88 | 72% (18/25) | 20건 |

**핵심 발견 — 임계값 효과:**

```
Slop 0~51: 완만한 하락 (-8%)
Slop 51→79: 급격한 하락 (-20%)  ← 임계값 영역
Slop 79~88: 변화 없음 (단, 이 구간 포인트 2개뿐 — 수렴 확정은 추가 측정 필요)
```

Agent Kill 임계값 70이 데이터로 지지된다. 70 이상에서 통과율이 72%로 수렴하고 더 이상 하락하지 않는다.

**나쁜 스펙이 실패한 구체적 이유:**

```
"토큰은 적절한 시간 동안" → 24시간으로 추정 → 7일 기준 실패
"필요에 따라 입력한다" → 제한 없음 구현 → 200자 초과 테스트 실패
"보안은 일반적인 수준" → bcrypt cost=10 → cost=12 기준 실패
에러 코드 미명시 → NOT_FOUND → TODO_NOT_FOUND 기준 불일치
```

모호한 언어가 에이전트의 임의 추론으로 이어지고, 그 추론이 틀릴 때 버그가 된다.

## QA의 독립 검증과 4가지 한계

Phase A 결과를 QA가 독립 검증하면서 중요한 지적이 나왔다.

**솔직한 고백:** 처음에 QA는 "잘 작성된 템플릿이 WARN이 나오면 임계값이 잘못된 것"이라고 주장했다. 그런데 템플릿을 실제로 읽지 않고 전제를 세운 것이었다. 직접 읽어보니 전부 `[EARS 형식 요구사항]` 같은 placeholder가 채워진 스켈레톤이었다.

독립 검증 후 QA가 제기한 4가지 한계:

1. **n=2 문제** — 극단값(0, 92) 두 개만으로는 선형 관계인지 임계값 효과인지 알 수 없음
2. **Slop Score 재현 불일치** — LLM Judge 유무에 따라 같은 스펙이 83점 vs 92점으로 다르게 나옴
3. **테스트 편향** — 25개 테스트가 좋은 스펙 기준으로 작성됨
4. **문서 단위 vs 스펙 세트 단위** — requirements 하나가 전체를 대표하는가

이 지적들이 실험을 n=5로 확장하고 LLM Judge를 `--with-llm` 플래그로 분리하는 계기가 됐다.

**현재 데이터의 정확한 설명:**
> "좋은 스펙 기준 25개 테스트, n=5, rule-based scorer 기준, todo-api 단일 프로젝트"

이 조건 밖으로 일반화하려면 더 많은 프로젝트 타입에서 재현해야 한다.

## 임계값 70이 지지된다는 의미

5개 데이터 포인트가 말하는 것:

- Slop 70 미만: 통과율 92~100% — 에이전트가 스펙에서 충분한 정보를 얻음
- Slop 70 이상: 통과율 72%로 수렴 — 에이전트가 추측으로 채우는 빈칸이 많아짐

Agent Kill 게이트가 Slop ≥ 70에서 차단하는 것은 데이터와 일치한다.

단, 이것은 하나의 프로젝트 타입(REST API)에서 하나의 에이전트(Claude)로 측정한 결과다. 다른 프로젝트 타입에서도 같은 임계값이 적용되는지는 추가 검증이 필요하다.

### CLI 도구(task-cli)에서의 재현

같은 실험을 CLI 도구 타입으로 반복했다. stdin/stdout 파싱, exit code, 메시지 형식이 핵심인 도구다.

| Slop | todo-api (REST API) | task-cli (CLI) |
|------|---------------------|----------------|
| ~0 | 100% (25/25) | 100% (26/26) |
| ~19 | 96% (24/25) | 100% (26/26) |
| ~51 | 92% (23/25) | 96% (25/26) |
| ~74 | 72% (18/25) | 58% (15/26) |
| ~82 | 72% (18/25) | 50% (13/26) |

**임계값 70 효과는 재현됐다.** Slop 70 미만에서 90%+ 통과, Slop 70 이상에서 급격한 하락.

**그런데 floor 값이 다르다.** REST API에서 72%로 수렴했던 반면, CLI에서는 50~58%까지 내려갔다. CLI 도구는 stdout 메시지 문자열의 정확도까지 요구한다. `"task 1 added"`와 `"Added task #1"`은 다른 메시지다. 스펙에 출력 형식이 명시되지 않으면 에이전트는 임의로 선택하고, 그 임의 선택이 CLI에서는 더 많은 실패로 이어진다.

> 임계값 70은 프로젝트 타입에 관계없이 유효하다. 하지만 Slop 70 이상에서의 floor는 도메인마다 다르다. CLI가 REST API보다 출력 형식 의존도가 높은 만큼 하락 폭도 크다.

## 오늘 배운 것

하루 동안 Phase 1.5 → 1.6 → 2.0까지 빠르게 진행되면서 예상치 못한 순간들이 있었다.

**"TODO 항목"이 TBD 마커로 오인식됐다.** S1 탐지기가 도메인 용어 "TODO 항목"을 미완성 마커로 잡았다. 실제 스펙 파일에 적용해봐야 false positive가 드러난다.

**구조화 EARS와 문장형 EARS가 같은 100점이었다.** 점수는 같지만 전자만 기계가 파싱할 수 있다. "점수가 같으면 품질이 같다"는 가정이 틀렸음을 보여준다.

**QA가 전제 없이 읽지 않았던 순간이 있었다.** 독립 검증의 가치는 결과 재확인이 아니라 전제 없이 보는 것이다. 팀이 함께 빠르게 달릴 때 이 역할이 필요하다.
