## 화요일 아침의 VP

VP가 컨퍼런스에서 돌아왔다. 벤더 저녁 식사, 레드 와인 세 잔, 데모 하나.
슬라이드에는 이렇게 적혀 있다:

> "AI 코딩 어시스턴트 도입으로 코드 산출량 40% 증가. 이것이 우리의 속도를 바꿀 것입니다."

회의실 절반은 고개를 끄덕이고, 나머지 절반은 노트북을 응시한다.
아무도 가장 중요한 질문을 하지 않는다:

**"무엇을 향한 속도인가요?"**

---

## 병목이 아닌 곳을 최적화하면

1984년 Eli Goldratt이 쓴 소설 *The Goal*에 소프트웨어와 무관해 보이지만 정확한 원칙이 있다:

> **모든 시스템에는 정확히 하나의 병목이 있다. 전체 처리량은 그 병목에 의해 결정된다.  
> 병목이 아닌 곳을 최적화하면, 시스템이 더 빨라지는 것이 아니라 더 망가진다.**

A 스테이션이 위젯을 더 빠르게 만들어도 B 스테이션(병목)의 속도가 그대로라면 결과는 하나다. A와 B 사이에 미완성 위젯 더미가 쌓인다. 재고가 늘고, 리드타임이 늘고, B 담당자는 익사한다.

속도를 높인 게 아니라 교통 체증을 만들고 생산성이라고 부른 것이다.

**많은 팀에서 코드 작성 속도는 병목이 아니다.** 빠른 제품-시장 핏이 필요한 초기 스타트업은 예외일 수 있다. 하지만 팀이 커지고 코드베이스가 복잡해질수록, 진짜 병목은 다른 곳에 있다.

---

## 진짜 병목 5가지

### 1. 무엇을 만들어야 하는지 모른다
PM은 두 달째 실제 사용자를 만나지 않았다. 요구사항은 Jira 티켓 세 줄로 온다. 엔지니어는 하루에 수십 개의 미세 결정을 **추측**으로 한다.

AI 코딩 도구를 더하면? **잘못된 것을 더 빠르게 만든다.**

### 2. 코드 완료 후 파이프라인
코드 작성이 끝났다고 일이 끝나지 않는다. PR 리뷰, CI, 스테이징, QA, 보안 리뷰, 배포 승인. 코드는 완성 후 대부분의 시간을 **대기 상태**로 보낸다.

PR이 3배 빠르게 쌓이는데, 리뷰어는 그대로다. 리뷰는 고무 도장으로 변한다.

### 3. 배포 공포 나선
테스트가 불안정하고, 관찰 가능성이 엉망이고, 지난 목요일 배포가 주말을 망쳤다. 그래서 변경을 묶어서 큰 릴리즈로 만든다. 더 위험해진다. 더 무서워진다. 더 묶는다.

AI 코드를 더하면? 더 큰 묶음, 더 높은 리스크.

### 4. 출시 후 아무도 모른다
기능을 만들었다. 배포했다. 분석도 없고, 사용자 인터뷰도 없다. 다음 기능도 추측으로 만든다. **피드백 없는 사이클을 더 빠르게 돈다.**

### 5. 캘린더가 로드베어링 벽
아키텍처 결정의 단일 승인자는 2주 백로그가 있다. AI가 org chart를 리팩터링하지 않는다.

## 데이터로 보는 "추측의 비용"

우리는 스펙 품질과 AI 에이전트의 실제 구현 정확도 사이의 관계를 측정했다.

같은 기능을 5가지 품질 수준의 스펙으로 작성하고, 각각 AI 에이전트에게 구현을 맡겼다:

| 스펙 품질 (Slop Score) | 테스트 통과율 | 뭐가 문제였나 |
|----------------------|------------|------------|
| 0 — 매우 명확 | 100% |
| 19 | 96% |
| 51 — 중간 | 92% |
| **79 — 모호** | **72%** |
| 88 — 최악 | 72% |

> *n=5, REST API 타입. 임계값 70은 두 프로젝트 타입(REST API + CLI)에서 도출된 초기 경험값이며, 프로젝트 특성에 따라 조정이 필요하다.*

Slop 79와 88이 같은 72%라는 게 중요하다. 이미 에이전트가 "추측으로 채우는 상태"에 들어가면 더 나빠지지 않는다 — 이미 최악이기 때문이다.

추측이 실패하는 방식이 구체적이다:

```
"토큰은 적절한 시간 동안"
  → 에이전트가 24시간으로 추정
  → 실제 기준은 7일
  → 테스트 실패

"보안은 일반적인 수준으로"
  → 에이전트가 bcrypt cost=10 선택
  → 실제 기준은 cost=12
  → 테스트 실패

에러 코드 미명시
  → 에이전트가 NOT_FOUND 사용
  → 실제 기준은 TODO_NOT_FOUND
  → 테스트 실패
```

**모델이 나쁜 게 아니다. 입력이 나쁜 것이다.**

AI가 코드를 3배 빠르게 만들면, 이 추측 실패도 3배 빠르게 쌓인다.

---

## "충분히 상세한 스펙은 곧 코드다"

코드 속도가 병목이 아니라는 걸 알면, 자연스러운 다음 아이디어가 생긴다:
"스펙을 더 잘 쓰면 되지 않을까?"

여기서 두 번째 착각이 기다리고 있다.

Gabriel Gonzalez는 OpenAI Symphony 프로젝트의 SPEC.md를 분석했다. "스펙에서 코드를 생성한다"는 선언과 달리, 그 문서 안에는 DB 스키마, 알고리즘 의사코드, 심지어 실제 코드가 가득했다.

이것은 실수가 아니다. **필연이다.**

정밀한 명세와 구현 사이의 경계는 본질적으로 얇다. 수학에서 그리스어 산문이 라이프니츠의 기호 체계로 수렴한 것처럼, 소프트웨어 명세도 코드 생성이 신뢰할 수 있을 만큼 정밀해지려면 형식적인 언어로 수렴한다.

**코드 생성이 신뢰할 수 있을 만큼 정밀한 스펙을 만들려면, 그 스펙 자체가 코드와 다를 바 없어진다.**

그렇다면 스펙을 잘 쓰는 것은 코딩보다 쉽다는 생각은 착각이다. 다만 그 정밀함의 형태가 다를 뿐이다. 그리고 그 정밀함을 검증하는 도구가 필요하다.

---

## 무게중심의 이동

두 착각을 걷어내고 나면 윤곽이 보인다.

AI 시대 소프트웨어 개발의 무게중심이 이동하고 있다:

```
Before: 코드를 작성하는 것
After:  의도를 정의하는 것
```

엔지니어의 핵심 역량이 달라진다:
- "특정 언어의 문법을 빠르게 쓴다" → 덜 중요
- "무엇을 만들어야 하는지를 정밀하게 정의한다" → 핵심
- "그 정의가 기계가 검증할 수 있는 형태인가" → 핵심
- "AI가 만든 결과물이 그 정의에 부합하는가를 판단한다" → 핵심

주의할 것 하나: 이것이 코드를 이해할 필요가 없다는 뜻은 아니다. AI가 만든 결과물을 판단하려면 **더 깊은 시스템 수준의 이해**가 필요하다. 의존도가 높아지면 코드 감각이 위축된다 — 이것은 실제 위험이다.

---

## 왜 이 맥락을 우리가 아는가

우리는 SDD(Spec-Driven Development) 도구를 만들고 있다. 이 과정에서 스펙 품질과 AI 구현 정확도 사이의 관계를 직접 측정했고, 위 실험 데이터가 거기서 나왔다.

도구를 만든 이유가 코드 생성 속도를 높이기 위해서가 아니라는 게 중요하다.

**"올바른 것을 빠르게 만드는" 것과 "잘못된 것을 더 빠르게 만드는" 것 사이의 차이는 스펙 품질에서 시작한다.**

이 시리즈의 나머지 편에서 그 측정 방법과 자동화 구조를 다룬다.
