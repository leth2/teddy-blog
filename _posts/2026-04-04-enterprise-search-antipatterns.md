> 이 글은 1~5편과 다르다. 성공한 것들이 아니라 **실패하거나 역효과가 난 것들**을 분석한다.  
> 연구에서 "안 되는 것"을 발견하는 것도 "되는 것"만큼 가치 있다.

---

## 왜 실패 분석인가

5편에 걸쳐 Zulip + Saleor 두 코드베이스, 5가지 검색 방식, 파일 예측 + 탐색 두 태스크를 측정했다. 최종 SOTA는 Hybrid F1=0.477이었다.

하지만 그 과정에서 **직관적으로는 맞는 것 같았지만 실제로는 역효과였던 시도들**이 4가지 있었다. 이 실패들을 분석하면 "왜 AI 코드 검색이 어려운가"에 대한 더 깊은 이해가 생긴다.

---

## 실패 1: 도메인 지식 주입 (v4)

### 무엇을 했나

v3(ripgrep + AI)에 추가로 코드베이스 구조 지식을 주입했다. 각 디렉토리의 주요 파일별 docstring을 컨텍스트로 제공: "이 파일은 이런 역할을 한다."

### 결과

| | v3 | v4 (+ 도메인 지식) |
|-|----|-----------------|
| Zulip n=10 | 0.430 | 0.355 |
| Zulip n=30 | **0.345** | **0.269** |

n=10, n=30 모두에서 도메인 지식을 넣었을 때 F1이 낮아졌다.

### 왜 실패했나

**핵심 원인: Zulip은 공개 프로젝트다.**

Claude는 이미 Zulip 코드베이스를 학습 데이터로 알고 있다. 통제 실험:
- AI 단독 (힌트 없음): F1=0.204
- AI 단독 (Zulip 구조 힌트 추가): F1=0.211

힌트 유무 차이가 0.007. AI는 이미 알고 있다.

우리가 제공한 docstring이 AI의 기존 지식과 충돌하거나, AI가 "이 컨텍스트를 따라야 한다"고 판단해서 오히려 잘못된 방향으로 이끌렸다.

### 교훈

> **공개 프로젝트에 대한 학습된 모델에 도메인 지식을 추가하면 역효과가 날 수 있다.**  
> 모델이 이미 알고 있는 것과 우리가 제공하는 것이 충돌할 때, 모델은 혼란스러워진다.  
> 비공개 내부 코드베이스라면 결과가 달라질 수 있다.

Agentless 논문(2024)도 같은 결론을 낸다: "더 많은 컨텍스트 ≠ 더 좋은 성능."

## 실패 2: C-det (결정론적 그래프) — n=10의 거짓 신호

### 무엇을 했나

Python import 관계로 방향 그래프를 만들고, ripgrep으로 찾은 후보 파일들의 그래프 연결 수(in-degree)를 기준으로 AI 선택 없이 상위 N개를 반환했다.

"많이 import되는 파일 = 중요한 파일 = 수정할 파일" 가정.

### 결과

| | n=10 F1 | n=30 F1 |
|-|---------|---------|
| C-det | **0.292** | **0.102** |

n=10에서 0.292로 AI 단독(0.204)보다 높았다. 하지만 n=30에서 0.102로 폭락했다.

### 왜 실패했나

**핵심 오류: "중요한 파일 ≠ 수정할 파일"**

연결 수가 높은 파일 = 많은 곳에서 import하는 인프라 파일.  
실제 수정하는 파일 = 특정 기능을 담당하는 전문화된 파일.

이 두 집합은 거의 겹치지 않는다.

n=10에서 높게 나온 것은 순전히 운이었다. 10개 커밋 중 몇 개가 우연히 인프라 파일을 건드리는 커밋이었을 뿐이다. n=30으로 확장하자 진짜 성능이 드러났다.

### 교훈

> **소규모 샘플에서의 좋은 결과를 믿지 마라.**  
> n=10 C-det 0.292는 신호가 아니라 노이즈였다.  
> n을 30으로 늘렸을 때 0.102로 내려갔다. n=30도 여전히 작다.  
> 중요한 주장은 충분한 n으로 재현되어야 한다.

## 실패 3: 적응형 라우팅 — 틀린 가정

### 무엇을 했나

커밋 메시지에 함수명/클래스명이 있는지 감지하고:
- 함수명 있음 → RAG 사용 (의미론적 검색이 유리할 것)
- 없음 → v3 사용

"함수명이 명시되면 RAG가 해당 함수를 더 잘 찾을 것"이라는 가정.

### 결과

| | v3 | Adaptive |
|-|----|---------| 
| Zulip | **0.345** | **0.274** |
| Saleor | **0.146** | **0.213** |

Zulip에서 오히려 낮아졌다. (Saleor는 소폭 개선됐으나 통계적으로 유의미하지 않음.)

### 왜 실패했나

**가정이 반대였다.**

커밋 메시지에 함수명이 있으면, ripgrep으로 그 이름을 직접 검색하는 게 더 정확하다.

```
커밋: "Fix list_shipping_methods_for_checkout"
ripgrep으로 "list_shipping_methods_for_checkout" 검색 → 파일 직접 발견
RAG로 의미 검색 → 비슷한 이름들이 여러 개 나와서 혼란
```

RAG의 실제 강점은 함수명 매칭이 아니라 **키워드가 없을 때 의미론적 보완**이다.

함수명이 명시적으로 있을수록 오히려 ripgrep이 더 정밀하다.

### 교훈

> **"언제 어떤 도구가 유리한가"에 대한 직관이 틀릴 수 있다.**  
> 실험 없이 라우팅 규칙을 설계하면 잘못된 방향으로 오히려 성능을 깎는다.  
> RAG는 "보완재"지 "대체재"가 아니다.

## 실패 4: Co-change 패턴 확장 — 희소성의 벽

### 무엇을 했나

git 히스토리 500개 커밋에서 "같이 변경된 파일 쌍"의 빈도를 카운트한 co-change 매트릭스를 구축했다. Hybrid가 예측한 파일 X에서 X와 자주 함께 변경된 파일 Y도 후보에 추가했다.

직관: 과거에 함께 바뀐 파일들은 미래에도 함께 바뀔 가능성이 높다.

### 결과

| | Hybrid (SOTA) | Hybrid + Co-change |
|-|--------------|-------------------|
| Zulip | **0.477** | **0.320** |
| Saleor | **0.228** | **0.165** |

두 코드베이스 모두에서 오히려 낮아졌다.

### 왜 실패했나

**두 가지 근본 문제:**

**① 매트릭스가 너무 희소하다.**
- Zulip: 500커밋에서 의미 있는 co-change 쌍 40개 파일
- 예측된 파일 대부분이 co-change 이웃이 없음
- 이웃이 없으면 확장이 일어나지 않고, 있어도 1~2회 공동 변경은 신뢰할 수 없는 신호

**② 캐스케이딩 실패 문제.**
- Hybrid가 이미 예측을 못 한 케이스(F1=0) → co-change 확장해도 아무 이웃도 없거나 틀린 이웃만 추가됨
- "틀린 예측에서 틀린 확장" 패턴

Co-change는 **아키텍처 분석**(어떤 모듈이 결합되어 있나?)에는 유용하지만 **per-commit 예측**에는 적합하지 않다. 스케일이 다른 문제다.

### 교훈

> **직관적으로 맞는 아이디어도 스케일이 안 맞으면 작동하지 않는다.**  
> Co-change가 유용하려면 수천 개 커밋의 히스토리와 안정적인 코드베이스가 필요하다.  
> 500커밋, 빠르게 변하는 오픈소스에서는 신호 대 노이즈 비율이 너무 낮다.

## 메타 교훈: 엔지니어링 버그가 연구를 오염시킨다

위 4가지와 별개로, 연구 과정에서 **경로 정규화 버그**가 있었다.

- 하이브리드 (버그 있을 때): F1=0.113
- 하이브리드 (버그 수정 후): F1=0.477

4배 차이. 버그 하나가 "하이브리드는 효과 없다"는 잘못된 결론을 낼 뻔했다.

Chroma가 `/workspace/codebase/zerver/lib/markdown/__init__.py`로 인덱스를 저장했는데, ripgrep은 `zerver/lib/markdown/__init__.py`로 반환했다. 두 소스의 후보를 합칠 때 deduplication이 작동하지 않아서 같은 파일이 두 번 들어가고 후보 품질이 나빠졌다.

### 교훈

> **실험 코드의 엔지니어링 품질이 연구 결론을 결정한다.**  
> "이 방식이 안 된다"는 결론을 내리기 전에 구현 버그가 없는지 확인해야 한다.  
> 경로, 문자열 정규화, 데이터 포맷 불일치는 AI 연구에서 자주 발생하는 무음 실패다.

## 실패들이 가르쳐준 것 — 종합

| 실패 | 핵심 교훈 |
|------|----------|
| 도메인 지식 주입 | 모델이 이미 아는 것과 충돌하면 역효과 |
| C-det n=10 신호 | 소규모 샘플의 좋은 결과 ≠ 진짜 성능 |
| 적응형 라우팅 | RAG의 강점을 잘못 이해했음 |
| Co-change 확장 | 아이디어는 맞지만 스케일이 안 맞음 |
| 경로 정규화 버그 | 엔지니어링 버그가 연구 결론을 오염시킴 |

**공통 패턴:** 모두 "직관적으로 맞는 것 같았다." 실험 없이는 알 수 없었다.

## 마무리

연구에서 실패는 정보다. 이 5가지 실패가 없었다면 "왜 AI 코드 검색이 어려운가"에 대한 이해가 훨씬 얕았을 것이다.

- 도메인 지식 주입 실패 → 모델의 사전 지식과 제공 정보의 충돌 이해
- C-det 실패 → "중요한 파일 ≠ 수정할 파일" 이해
- 적응형 라우팅 실패 → RAG와 ripgrep의 실제 강점 이해
- Co-change 실패 → 스케일 의존성 이해
- 경로 버그 → 구현 품질의 중요성 재확인

다음에 비슷한 실험을 설계할 때 이 목록이 체크리스트가 될 것이다.
